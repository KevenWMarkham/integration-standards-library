ISL Implementation Accelerator

Detailed operations manual for deploying the Integration Standards Library at client sites.

What It Does

Transforms the ISL's 39 standards templates into client-specific, platform-ready artifacts. Consultants configure a client profile, select standards modules, enable platform adapters, and generate deployment-ready outputs in a single workflow.

Key Specs

  • 26 files, zero external dependencies
  • Pure JavaScript + HTML, runs from filesystem
  • Two-layer adapter engine (Context + Platform)
  • 4 built-in adapters + plugin extensibility
  • 3 industry profiles + blank template
  • ZIP export with deployment scripts
i
The accelerator is designed to reduce Phase 4 (Standards Definition) effort by 30-57% per engagement, saving 200-400 hours across the 6 ISL modules.

Qualitative Benefits

Consistency

Without the accelerator, 3-5 consultants independently draft standards, leading to conflicting terminology, inconsistent naming patterns, and misaligned classification tiers across modules. The accelerator guarantees cross-module consistency through a single resolved configuration.

Client Confidence

Clients see polished, production-quality documents from the first review — not rough drafts. Industry-specific language (ITAR/EAR for manufacturing, PCI-DSS for financial services, HIPAA for healthcare) is built into profiles. This signals domain expertise from day one.

Repeatability

Every engagement follows the same proven structure. New consultants ramp up in hours (run wizard, review output) instead of weeks (learn ISL, draft from scratch). Configuration is exportable JSON — reusable across engagements in the same industry.

Scope Control

Without the accelerator, scope creep is common: "Can we also add data lineage?" means 30-50 hours of new work. With the accelerator, adding a module is a checkbox in the wizard and a re-generate — the marginal cost of scope expansion drops dramatically.

Quick Start

  1. Open the accelerator — Double-click implementation/index.html in your browser. No server required.
  2. Complete the wizard — First-time users see a 5-step setup wizard. Enter client name, select industry profile, choose modules, enable adapters.
  3. Generate outputs — Click Generate on the review screen. The engine transforms templates, runs adapters, and bundles everything.
  4. Download the ZIP — Navigate to the Deploy tab and download the output package containing adapted standards, platform configs, and deployment scripts.
  5. Deploy at client site — Run the generated PowerShell scripts (e.g., Deploy-FabricNaming.ps1) against the client's Azure environment. Use -WhatIf first to preview.

Architecture

The accelerator uses a three-layer architecture: Presentation (Wizard + Dashboard), Adapter Engine (Context + Platform), and Output Generator.

Presentation Layer Wizard (5-step) | Dashboard (ongoing) | Module Configurator | Deploy Console ───────────────────────────────────────────────────────────────────────────────── Adapter Engine Layer 1: Client Context ProfileLoader ──> ParameterResolver ──> TemplateTransformer (industry profile) (3-layer merge) ({{placeholder}} substitution) Layer 2: Platform Adapters FabricAdapter | PurviewAdapter | ApimAdapter | WebhookAdapter + PluginRegistry (extensible) ───────────────────────────────────────────────────────────────────────────────── Output Generator Adapted Markdown | Platform Configs (JSON/XML) | Deploy Scripts (PS1/Bash) Audit Manifest | ZIP Package Builder

Data Flow

  1. Profile baseline — Industry profile (Manufacturing, Financial Services, Healthcare) provides sensible defaults for naming, classification, regulatory frameworks.
  2. Client overrides — Consultant edits fields in the wizard or module configurator. Overrides merge on top of the profile.
  3. Template transformation — The TemplateTransformer substitutes {{dot.path}} placeholders with resolved config values, processes conditionals and loops.
  4. Platform adaptation — Each enabled adapter converts transformed templates into platform-native artifacts (Fabric JSON, Purview policies, APIM XML, etc.).
  5. Output bundling — All artifacts are bundled into a structured ZIP with configs, docs, scripts, and an audit manifest.

Merge Priority

Profile Baseline  <  Client Overrides  <  Per-Template Overrides
(lowest priority)                        (highest priority)

Each layer wins over the previous. A consultant can override a single naming separator without re-specifying the entire Manufacturing profile.

Use Cases

Greenfield Fabric Deployment
Client standing up Microsoft Fabric for the first time. ISL-03 establishes workspace, lakehouse, warehouse, pipeline, and semantic model naming before any assets are created — preventing the naming chaos that typically requires expensive refactoring 6 months later.
Synapse-to-Fabric Migration
Client migrating from Azure Synapse. ISL-03 provides a migration naming map — existing Synapse artifacts get renamed to the ISL standard during migration, eliminating technical debt carry-over and establishing naming discipline from day one in Fabric.
Multi-Team Consistency Enforcement
Large enterprise with 5+ data engineering teams creating assets independently. Each team uses different abbreviations and patterns. ISL-03's abbreviation dictionary and naming formulas create a single source of truth, enforced via Azure Policy and pre-commit hooks.
Dual-ERP Manufacturer
Client running SAP S/4HANA and Epicor Kinetic across different facilities. ISL-03 provides standardized domain codes, ERP module abbreviations, and IoT source identifiers that unify naming across both ERPs in a single lakehouse, preventing confusion between sap_pp and epc_prod work orders.

Industry Segmentation & Regulations

The accelerator's core value proposition is industry-aware standards generation. Rather than starting every engagement from a blank page, the consultant selects an industry profile that pre-configures regulatory frameworks, classification tiers, data domains, naming patterns, and platform security controls. This section explains why segmentation matters, how the regulatory landscape drives configuration decisions, and how profiles encode that knowledge.

Why Industry Segmentation Matters

Data governance is not one-size-fits-all. A defense manufacturer handling ITAR-controlled technical drawings has fundamentally different classification requirements than a hospital managing PHI under HIPAA or a bank processing cardholder data under PCI-DSS. The differences are not cosmetic — they affect:

  • Classification tier structure — Financial services typically need 5 tiers (adding "Highly Confidential" between Confidential and Restricted), while manufacturing and healthcare use 4 tiers with industry-specific labels (e.g., "Confidential-PHI")
  • Data domain taxonomy — A manufacturer's data domains (mfg, scm, qms, iot) share zero overlap with a hospital's (cln, phm, ehr, gnm) or a bank's (rtl, trd, rsk, pay)
  • Regulatory controls — ITAR requires US Person access verification and data residency; HIPAA requires minimum necessary access and breach notification; PCI-DSS requires network segmentation and key rotation
  • Platform configuration — Purview labels, Fabric workspace isolation, APIM rate limits, and deployment scripts all vary by regulatory context
  • Special data categories — Each industry has protected data classes that don't exist in other industries (CUI for defense, PHI for healthcare, Material-Non-Public for finance)
!
Without segmentation, consultants spend 40-60 hours in discovery interviews asking the same questions that have well-known answers for each industry. The accelerator eliminates this by encoding industry knowledge into profiles, so interviews focus only on what's unique to this specific client.

Naming Standardization Philosophy

ISL-03 is the foundational module of the entire library. Every other module references naming conventions — from ISL-04 workspace classification suffixes to ISL-05 pipeline patterns to ISL-06 quality rule identifiers. This section explains the why behind the naming framework: why consistency matters at scale, why the specific pattern structure was chosen, and how naming governance is enforced.

Why Naming Matters

Cognitive Load Reduction

In a Fabric lakehouse with 500+ tables across 12+ domains, consistent naming is the primary navigation mechanism. When a developer sees slv_mfg_work_order_current, they immediately know: silver layer, manufacturing domain, work order entity, current-state SCD. No documentation lookup needed. Studies show that consistent naming reduces onboarding time from 2–3 days to less than 1 day.

Automation Enablement

Naming conventions enable programmatic operations. Azure Policy can enforce resource naming via deny rules. CI/CD pipelines can validate artifact names against the abbreviation dictionary before deployment. Purview scanning can auto-classify assets by parsing tier indicators from workspace names. Without standard names, automation requires expensive custom mapping tables.

Cross-Team Communication

When the data engineering team, the analytics team, the security team, and the client's business users all use the same naming vocabulary, handoff friction drops dramatically. The abbreviation dictionary serves as a shared language — mfg always means Manufacturing, fin always means Finance, brz always means Bronze layer.

Pattern Rationale: {prefix}_{domain}_{entity}_{qualifier}

The ISL-03 naming pattern uses a segmented structure with underscores as delimiters. Each segment carries specific semantic meaning and position-dependent interpretation:

Position Segment Purpose Examples
1 Resource prefix Identifies artifact type at a glance ws, lh, pl, nb, fct, dim
2 Domain / source Business domain or source system ownership mfg, fin, scm, sap, iot
3 Entity / purpose The core business concept or function work_order, analytics, bronze, quality_predict
4 Qualifier / version Environment, temporality, version, or frequency prd, v1, daily, current, snapshot

Data Classification Principles

ISL-04 defines a 4-tier classification model that serves as the security backbone for the entire standards library. This section explains the theoretical foundations: why the CIA triad maps to classification, why exactly four tiers, and how classification changes dynamically across the data lifecycle.

The CIA Triad and Classification

Every classification decision is ultimately a risk assessment across three dimensions: Confidentiality, Integrity, and Availability (CIA). The ISL-04 4-tier model maps primarily to confidentiality impact (what happens if the data is disclosed?), but tiers also implicitly set expectations for integrity and availability controls.

CIA Dimension Tier 1 — Public Tier 2 — Internal Tier 3 — Confidential Tier 4 — Restricted
Confidentiality None — public by design Low — minor inconvenience if disclosed Moderate — financial loss, regulatory fine, reputational damage Severe — existential threat, criminal liability, national security
Integrity Platform default; no special controls Standard checksums; annual review Change tracking; quarterly audit; hash validation Immutable storage; real-time integrity monitoring; crypto-verification
Availability Best effort; no SLA Standard platform SLA (99.9%) Enhanced SLA; 4-hour RPO; daily backup Mission-critical SLA; 1-hour RPO; real-time replication; DR tested

This mapping ensures that higher-tier data automatically receives proportionally stronger controls across all three dimensions — not just encryption and access control (confidentiality), but also backup, integrity monitoring, and availability guarantees.

Why Exactly Four Tiers

Too Few (2–3 Tiers)

A 2-tier model (public/private) forces all sensitive data into one bucket, applying the same expensive Tier 4 controls to routine PII and ITAR-controlled data alike. A 3-tier model (public/internal/restricted) still conflates moderate-sensitivity data (customer names) with extreme-sensitivity data (SSNs, trade secrets), leading to either over-protection of Confidential data or under-protection of Restricted data.

Too Many (5+ Tiers)

More than 4 tiers creates classification fatigue. Data stewards struggle to distinguish between 5 or 6 similar tiers, leading to inconsistent classification and governance overhead. Each additional tier requires its own handling rules, access patterns, encryption settings, and review cadences — the operational cost grows exponentially while the marginal security benefit diminishes.

The 4-Tier Sweet Spot

Four tiers align with industry standards (NIST SP 800-60 low/moderate/high + unrestricted), regulatory expectations, and human cognitive capacity. Each tier is clearly distinguishable: "anyone can see it" (Public), "employees can see it" (Internal), "specific teams need approval" (Confidential), "named individuals with JIT access" (Restricted). Sub-classifications within Tier 3 and Tier 4 handle regulatory nuance without adding tiers.

Dynamic Classification

Data classification is not static. The same data can change tier based on context, timing, and combination. ISL-04 defines explicit reclassification triggers and approval workflows.

Scenario Direction Example New Tier
Public disclosure Downgrade Quarterly financial results published in 10-K filing Tier 3 → Tier 1
M&A activity begins Upgrade Standard business data becomes material non-public information during deal Tier 2 → Tier 4
Contract expiration Downgrade NDA expires; formerly confidential vendor terms become internal-only Tier 3 → Tier 2
Data anonymization Downgrade PII is removed or tokenized; Confidential PII becomes Internal Tier 3 → Tier 2
Export control determination Upgrade Engineering data found to be ITAR-controlled after USML review Tier 3 → Tier 4
Security incident Upgrade Data integrity compromised; elevated controls during investigation Any → +1 Tier
Regulation change Either New privacy law covers previously unregulated data type Tier 2 → Tier 3
Data enrichment Upgrade Internal report joined with PII from HR system Tier 2 → Tier 3

Aggregate Classification

One of the most common classification errors is failing to reclassify aggregated data. ISL-04 defines the aggregation escalation rule: when individually non-sensitive data is combined in a way that reveals sensitive patterns, the aggregated dataset must be classified at the highest tier justified by the derived insight.

⚠ Aggregation example: Individual IoT temperature readings from a furnace are Tier 2 (Internal — routine operational data). But a dataset combining those readings with proprietary cycle times and alloy-specific parameters that reveals a trade-secret heat treatment process is Tier 4 (Restricted — Trade Secret). The individual data points are innocuous; the combination is the crown jewel.

Similarly, when a dataset, document, or container includes data elements at multiple tiers, the highest tier present governs the entire asset unless granular controls (column-level security, row-level security, or workspace isolation) isolate the higher-tier elements. A database table with 50 Tier 2 columns and 1 Tier 4 column (e.g., ssn) is a Tier 4 table unless column-level security masks the SSN column from non-authorized users.

Inherited Classification

Child datasets inherit the classification of their parent source unless a transformation provably removes the sensitive elements. This principle applies across the data pipeline:

Parent Asset Child Asset Inherited Tier Exception
Tier 3 source table Silver-layer cleansed table Tier 3 (inherited) Drops to Tier 2 only if all PII columns are removed or tokenized
Tier 4 ITAR dataset Gold-layer aggregated report Tier 4 (inherited) ITAR classification persists regardless of aggregation; no downgrade permitted without DDTC review
Tier 3 Fabric dataset Power BI report Tier 3 (inherited via sensitivity label propagation) Purview labels propagate automatically from datasets to dependent reports and dashboards
Tier 2 + Tier 3 join Resulting combined dataset Tier 3 (highest tier governs) No exception — highest tier always wins in a join
Any tier + Tier 4 join Resulting combined dataset Tier 4 (highest tier governs) Workspace must be Tier 4 or the join must be refactored to avoid co-mingling

Declassification Procedures

Lowering a classification tier requires formal approval to prevent accidental exposure. ISL-04 defines a tiered approval matrix based on the magnitude of the downgrade:

Current Tier Target Tier Approver(s) Process Typical Timeline
Tier 3 Tier 2 Data Owner + Governance Council Justification submitted; 30-day comment period; regulatory review 30–45 days
Tier 4 Tier 3 Data Owner + CISO + Legal Formal declassification review; regulatory clearance; legal sign-off 45–60 days
Tier 4 Tier 2 Data Owner + CISO + Legal Full declassification review; two-step downgrade (4→3, then 3→2) recommended 60–90 days
Any tier Tier 1 (Public) Data Owner + Communications + Legal Public release approval; regulatory check; competitive impact review; IP clearance 30–60 days
✓ Cardinal rule: "When in doubt, classify up." ISL-04 mandates that when the correct tier is ambiguous, the data must be classified one tier higher than the lowest plausible tier. It is always easier — and dramatically cheaper — to declassify data downward through the formal process than to recover from a breach caused by under-classification. Unclassified data defaults to Tier 2 (Internal), never Tier 1 (Public).

The Six Quality Dimensions — DAMA DMBOK Alignment

Measurement Formulas

Each dimension is quantified as a rate between 0% and 100% (except Timeliness, measured in minutes).

Dimension Formula (SQL-like) Edge Case

Composite Score: Score = (w1 × Completeness) + (w2 × Accuracy) + (w3 × Timeliness) + (w4 × Consistency) + (w5 × Validity) + (w6 × Uniqueness) where all weights default to 1/6 and sum to 1.0. Weights are configurable per domain and use case.

Statistical Process Control (SPC) for Data Quality

SPC techniques borrowed from manufacturing quality management help distinguish normal variation from assignable-cause degradation. The ISL-06 monitoring framework uses these concepts for regression detection and trend analysis.

Control Limits

  • Center line (CL): 30-day rolling mean of daily quality scores
  • Upper Control Limit (UCL): CL + 3σ
  • Lower Control Limit (LCL): CL − 3σ
  • Points outside LCL trigger a REGRESSION alert
  • Points between LCL and CL − 2σ trigger an ANOMALY warning

Run Rules (Western Electric)

  • Rule 1: Any single point beyond 3σ → out of control
  • Rule 2: 2 of 3 consecutive points beyond 2σ → warning
  • Rule 3: 4 of 5 consecutive points beyond 1σ → trend
  • Rule 4: 8 consecutive points on same side of CL → shift
  • When quality is "in control," variation is random and no action is needed

Threshold Calibration Methodology

  1. Baseline: Profile current data to establish actual null rates, duplicate rates, and format compliance percentages.

Common Validation Scenarios

Org prefix left blank
Wizard Step 5 shows error: "Missing required fields: naming.orgPrefix". All naming patterns resolve to _p_mfg_bronze instead of acme_p_mfg_bronze. Fix: Go back to Step 1, enter the 3-5 character org prefix.
Tier count says 5 but only 4 labels provided
Wizard Step 5 shows warning: "Tier count (5) does not match tier labels length (4)". Classification templates will reference a 5th tier that has no label, producing blank entries. Fix: Go to Step 3, either reduce tier count to 4 or add the 5th label.
No adapters enabled but modules selected
Wizard shows warning: "No adapters enabled". Templates will be transformed (placeholders resolved) but no platform-specific configs or deployment scripts will be generated. The output ZIP will contain only adapted Markdown standards — no JSON, XML, or PowerShell. Useful for standards-only engagements.

Troubleshooting

App shows "Loading accelerator..." spinner indefinitely

!
Cause: Opening index.html via file:// protocol with ES module scripts (legacy issue, now fixed).
Fix: Ensure you're using the latest index.html which loads scripts as classic <script src="..."> tags, not <script type="module">. The current version works on both file:// and http://.

Configuration lost after browser update

LocalStorage may be cleared by browser updates or privacy settings. Mitigate by exporting your config regularly (Dashboard → Export Config).

Fonts not loading (offline)

Google Fonts require internet access. The CSS falls back to system fonts (-apple-system, BlinkMacSystemFont, Segoe UI) when offline. The UI is fully functional without Google Fonts.

PowerShell scripts fail with auth errors

The generated deployment scripts require authenticated sessions:

  • Fabric: Run Connect-AzAccount first
  • Purview: Run Connect-MgGraph -Scopes "InformationProtectionPolicy.ReadWrite.All"
  • APIM: Run Connect-AzAccount and ensure the Az.ApiManagement module is installed

Always run scripts with -WhatIf first to preview changes before deploying.

ZIP download doesn't work

The ZIP builder uses the browser's Blob and URL.createObjectURL APIs. Ensure your browser supports these (all modern browsers do). If using an older browser, try Chrome or Edge.

Plugin not appearing in wizard

  • Verify the manifest.json has all required fields (id, name, version, modules, layer, entryPoint)
  • Verify the adapter class implements validate(), transform(), and generateScripts()
  • Register the plugin in your app code: pluginRegistry.register(manifest, AdapterClass)
